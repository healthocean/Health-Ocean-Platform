--------------------------------------------------------------------------------
CONCEPT
--------------------------------------------------------------------------------
Health Ocean is a diagnostic lab booking platform — think "Practo for lab tests".
Users can browse diagnostic tests and health packages offered by registered labs,
add them to a cart, book a home sample collection appointment, and track their
bookings and reports. Labs register on the platform, list their tests/packages,
and manage bookings via a dedicated portal. Admins oversee everything via a
separate admin panel.

Tagline: "Dive into Better Health"

API ROUTES SUMMARY
--------------------------------------------------------------------------------
POST   /api/users/register          — user signup
POST   /api/users/login             — user login
GET    /api/users/profile           — get user profile (JWT required)

GET    /api/tests                   — list all active tests (from all labs)
GET    /api/tests/:id               — single test detail

GET    /api/packages                — list all active packages
GET    /api/packages/:id            — single package detail

POST   /api/bookings                — create booking (auto-resolves labId from testIds)
GET    /api/bookings                — list bookings (filter by ?email=)
GET    /api/bookings/:id            — single booking
PATCH  /api/bookings/:id/cancel     — cancel booking

POST   /api/labs/register           — lab registration
POST   /api/labs/login              — lab login
POST   /api/labs/employees/register — register lab employee
GET    /api/labs/:labId/tests       — lab's own tests
POST   /api/labs/:labId/tests       — add test
GET    /api/labs/:labId/packages    — lab's own packages
POST   /api/labs/:labId/packages    — add package
GET    /api/labs/:labId/stats       — dashboard stats (bookings, revenue, top tests)
GET    /api/labs/:labId/bookings    — bookings for this lab's tests
PATCH  /api/labs/tests/:testId      — update test status
DELETE /api/labs/tests/:testId      — delete test
PATCH  /api/labs/packages/:id       — update package status
DELETE /api/labs/packages/:id       — delete package
GET    /api/labs/:labId             — get lab details

POST   /api/admin/login             — admin login
GET    /api/admin/labs              — list all labs (with status filter)
PATCH  /api/admin/labs/:labId/status — approve/reject lab
GET    /api/admin/users             — list all users
GET    /api/admin/stats             — platform-wide stats

DATABASE MODELS
--------------------------------------------------------------------------------
User:         name, email, phone, password (hashed)
Lab:          labId, name, email, phone, password, address, city, state, pincode,
              nablCertificate, isoCertificate, licenseNumber, establishedYear,
              description, servicesOffered[], status (Pending/Approved/Rejected)
LabEmployee:  employeeId, labId, name, email, phone, password, role
              (Admin/Manager/Technician/Receptionist)
LabTest:      testId (TEST<timestamp>), labId, name, description, category,
              price, originalPrice, sampleType, turnaroundTime,
              preparationRequired, preparationInstructions, parameters[], status
LabPackage:   packageId (PKG<timestamp>), labId, name, description, category,
              price, originalPrice, testsIncluded[], features[], status
Booking:      bookingId (BK<timestamp>), name, email, phone, address, city,
              pincode, date, timeSlot, testIds[], packageIds[], labId,
              couponCode, discount, total, totalAmount, status
              (Confirmed/InProgress/Completed/Cancelled), phlebotomistAssigned
Admin:        adminId, name, email, password, role (SuperAdmin/Admin)

IMPORTANT — testId mismatch fix:
  Mobile app sends MongoDB _id as testIds in bookings.
  Lab stats/bookings endpoints match on BOTH _id AND custom testId field.
  Booking creation auto-resolves labId from the first testId.

--------------------------------------------------------------------------------
BOOKING FLOW (end-to-end)
--------------------------------------------------------------------------------
1. User browses tests/packages → taps card → opens detail page
2. Taps "Add to Cart" → CartProvider stores item with _id, name, price
3. Goes to Cart → taps "Proceed to Book"
4. BookingScreen Step 1: Enter address + pincode (OSM lookup) + map location
5. BookingScreen Step 2: Pick date (TableCalendar) + time slot
6. BookingScreen Step 3: Review + apply coupon (HEALTH10/OCEAN20/FIRST15)
7. PaymentScreen: COD only currently → POST /api/bookings
   Booking stores testIds (as MongoDB _id strings), API resolves labId
8. Lab sees booking in /dashboard bookings table

🚀 2. Game-Changing Feature Ideas (To Make Health Ocean Unique)
To move beyond just being a "Practo clone" and offer an exceptional, sticky user experience, consider these advanced features:

🧠 A. AI-Powered Report Summaries (Massive Value Add)
The Problem: Patients receive complex PDFs full of medical jargon and reference ranges they don't understand, causing anxiety.
The Solution: Integrate an LLM (like Gemini or OpenAI) on the backend. When a lab uploads a PDF, the backend quietly parses the text, extracts out-of-bound markers, and generates a simple, plain-English "Health Summary" payload.
App Experience: Under the PDF download button, users see a “What this means” card highlighting key takeaways (e.g., "Your Vitamin D levels are slightly below normal, but your Lipid profile has improved since last time.").
📈 B. visual Smart Health History (Biomarker Graphing)
Instead of treating tests as isolated PDFs, require labs to input (or use OCR to extract) key metrics (HbA1c, Cholesterol, Thyroid).
Plot these metrics on beautifully interactive line graphs in the Health History screen. Users can visually track their health journey over the years.
👨‍👩‍👧‍👦 C. Family Member Profiles
Usually, one tech-savvy member of a household books tests for elderly parents or children.
Allow users to create sub-profiles under their main account. During booking, they select "Who is this test for?". The app automatically segregates reports and health histories by family member.
📍 D. Live Phlebotomist Tracking
Give the feeling of Uber/Swiggy to healthcare. If a home sample collection is booked, integrate Google Maps in the app to show the live location of the phlebotomist on the day of the appointment with an ETA.
💬 E. WhatsApp API Integration
In markets where WhatsApp is dominant, email isn't always checked quickly. Use a service like Twilio or Meta's WhatsApp API to automatically push the PDF report to the user’s WhatsApp as soon as the lab marks the booking as "Completed".

🔄 F. Preventative Care Subscriptions
Create recurring "Health Ocean Packages" (e.g., Quarterly Diabetic Checkup, Annual Comprehensive Wellness).
Users subscribe once via a saved card/UPI mandate. The app automatically schedules their home collections every X months, generating highly predictable recurring revenue for the platform.

🌙 G. Offline Report Vault
Use SQLite or Hive locally in Flutter to cache downloaded PDFs and the JSON of report summaries. Allow users to open their "Report Vault" at the doctor's clinic even if they have a bad mobile network connection.

🤖 H. Conversational Health Booking
Turn the Sanmare Assist chatbot into a booking engine. Allow users to say "Book a thyroid test for tomorrow morning at my home" and have the AI prepare the cart and navigate them directly to the review screen.

🎙️ I. Voice-First Interaction
Add a microphone button to Sanmare Assist for hands-free health queries, making the app accessible for elderly users or those with limited mobility.
